**Author:** Claude Opus 4.6 + Mike Gandal

**Status:** Draft
**Prerequisite:** Phase 1 (complete), Agent Architecture Redesign (complete)

## Overview

Phase 2 makes NanoClaw agents smarter in two ways: they can discover capabilities they don't have (Skill Discovery), and they preserve context across long sessions that would otherwise be lost to compaction (Lossless Conversation Memory).

**Scope:** Two features only. Proactive Bus Publishing (originally spec'd as 2.3) is deferred to the Ops Routing workstream per the Phase 1 spec's own recommendation.

**Dependencies resolved:**
- Agent Architecture Redesign: complete (identity.md, write_agent_memory IPC, /workspace/agents mount)
- Bus trust enforcement: already secure (sourceGroup is host-verified, unforgeable)

## Feature 2.1: Skill Discovery

### Problem

When an agent can't do something, it has no way to discover available capabilities. There are 20+ skill branches with rich SKILL.md files, but agents can't search them.

### Solution

A QMD-backed skill catalog generated by auto-scanning SKILL.md files from skill branches. Agents search the catalog via an MCP tool and tell the user how to install missing skills.

### Architecture

```
git skill/* branches → skill-catalog-sync.sh → data/skill-catalog/*.md → QMD collection → skill_search MCP tool → agent
```

### Components

#### 2.1.1 Skill Catalog Generator

**File:** `scripts/sync/skill-catalog-sync.sh` (new)

Host-side bash script that:
1. Runs `git branch -a | grep 'skill/'` to enumerate all skill branches
2. For each branch, reads `SKILL.md` from the branch tip (`git show {branch}:SKILL.md`)
3. Extracts YAML frontmatter: name, description, triggers
4. Checks installed status: `git merge-base --is-ancestor {branch} HEAD` (exit 0 = installed)
5. Writes `data/skill-catalog/{skill-name}.md` with standardized frontmatter:

```yaml
---
name: add-telegram
description: Add Telegram as a channel. Can replace WhatsApp or run alongside it.
triggers: ["add telegram", "telegram channel", "telegram setup"]
installed: true
install_command: "/add-telegram"
branch: skill/add-telegram
---

Add Telegram as a channel. Can replace WhatsApp entirely or run alongside it.
Also configurable as a control-only channel (triggers actions) or passive channel
(receives notifications only).

## Capabilities
- Telegram Bot API integration via Grammy
- Group and private chat support
- Agent swarm with per-bot identities
```

The body is the full SKILL.md content (minus frontmatter), which gives QMD rich text for semantic search.

**Integration:** Add as step 8 in `scripts/sync/sync-all.sh` (runs every 4 hours, after QMD update). The catalog refresh is cheap (git show is fast) and idempotent.

#### 2.1.2 QMD Collection

**Collection name:** `skill-catalog`
**Path:** `data/skill-catalog/`
**Pattern:** `**/*.md`

Added via `qmd add skill-catalog data/skill-catalog --ext md`. Indexed during the existing `qmd update` step in sync-all.sh. Embedded during `qmd embed`.

#### 2.1.3 MCP Tool: skill_search

**File:** `container/agent-runner/src/ipc-mcp-stdio.ts` (add tool)

```typescript
server.tool(
  'skill_search',
  'Search for NanoClaw capabilities and skills. Use when you need to do something ' +
    'but don\'t have the right tool. Returns matching skills with install instructions.',
  {
    need: z.string().describe('What you need to do (natural language)'),
  },
  async (args) => {
    // Delegate to QMD via IPC — host has QMD access
    writeIpcFile(TASKS_DIR, {
      type: 'skill_search',
      query: args.need,
    });
    // Wait for result (poll pattern, 10s timeout)
    const result = await pollForResult('skill_search', 10000);
    return { content: [{ type: 'text', text: result || 'No matching skills found.' }] };
  },
);
```

**Host-side IPC handler** (`src/ipc.ts`): New `skill_search` case that queries QMD `skill-catalog` collection with the agent's `need` string, formats results with name, description, installed status, and install command.

#### 2.1.4 No Auto-Install

When an agent finds an uninstalled skill, it tells the user directly via send_message:

> "I found a skill called *add-telegram* that can do this: 'Add Telegram as a channel.' To install it, run `/add-telegram`."

No approval workflow, no auto-install, no restart handling. User stays in full control.

## Feature 2.2: Lossless Conversation Memory

### Problem

Long agent sessions lose coherence when the SDK compacts (truncates) conversation context. Decisions, commitments, and learned facts vanish. The next session starts with no memory of what was discussed.

### Solution

When context compaction is triggered, the agent itself extracts key items from the conversation and writes them to its persistent memory via the existing `write_agent_memory` IPC action. The context assembler then injects these items into subsequent context packets.

### Architecture

```
SDK PreCompact hook fires → inject self-extract prompt → agent calls write_agent_memory →
items persisted to data/agents/{name}/memory.md (## Session Continuity) →
next spawn: context-assembler reads memory.md → injects continuity section into context packet
```

### Components

#### 2.2.1 PreCompact Hook Extension

**File:** `container/agent-runner/src/index.ts` (extend existing hook, lines 308-354)

The existing `PreCompact` hook:
- Fires before the SDK truncates conversation context
- Archives the full transcript to `group/conversations/{date}-{summary}.md`

**Extension:** After archiving, inject a system-level prompt into the session that instructs the agent to extract and persist its key context:

```typescript
// After transcript archiving (existing), add:
const extractPrompt = `[SYSTEM] Your conversation context is about to be compacted. ` +
  `Before it's lost, call write_agent_memory with a concise summary of:\n` +
  `- Decisions made in this session\n` +
  `- Tasks you committed to\n` +
  `- Open questions or unresolved items\n` +
  `- Key facts the user shared that you'll need later\n` +
  `Use bullet points. Write to section "Session Continuity". Be concise.`;
```

The hook then waits up to 30 seconds for the agent to process this prompt and call `write_agent_memory`. If the timeout expires, compaction proceeds without extraction (fail-open — we don't block the session).

**Implementation detail:** The PreCompact hook returns a Promise. The SDK waits for it to resolve before compacting. We resolve after either:
- The agent writes to `write_agent_memory` (detected via IPC output file watch), or
- 30 seconds elapse (timeout)

#### 2.2.2 write_agent_memory Section Handling

**Existing IPC action** (already implemented in Phase 1 architecture redesign):
- Agent calls `write_agent_memory` with content
- Host writes to `data/agents/{name}/memory.md`

**New behavior:** When the content targets `## Session Continuity`:
- Replace the existing `## Session Continuity` section (if any) rather than appending to the whole file
- Cap at 2000 characters (truncate oldest items if exceeded)
- Clear at session start: when a new session begins (not a compaction continuation), remove the `## Session Continuity` section from memory.md

**No changes to IPC handler needed** — the agent writes the full content including the section header, and `write_agent_memory` already replaces content.

#### 2.2.3 Context Packet Injection

**File:** `src/context-assembler.ts` (add section)

Add a new section after the agent identity block (for compound groups) or after group memory (for simple groups):

```typescript
// Read agent memory.md for Session Continuity section
const agentMemoryPath = path.join(DATA_DIR, 'agents', agentName, 'memory.md');
if (fs.existsSync(agentMemoryPath)) {
  const memoryContent = fs.readFileSync(agentMemoryPath, 'utf-8');
  const continuityMatch = memoryContent.match(
    /## Session Continuity\n([\s\S]*?)(?=\n## |\n$|$)/,
  );
  if (continuityMatch?.[1]?.trim()) {
    const continuity = continuityMatch[1].trim().slice(0, 1500);
    sections.push(`## Session Continuity (from prior compaction)\n${continuity}`);
  }
}
```

**Budget:** 1500 characters max. The current context packet is well under the 16KB limit after the architecture redesign reduced CLAUDE.md sizes by ~60%.

#### 2.2.4 Session Start Cleanup

**File:** `src/container-runner.ts` (host-side, before container spawn)

When a new session starts (not a continuation after compaction):
- Check if `data/agents/{name}/memory.md` has a `## Session Continuity` section
- If so, remove it (the context from the prior session is stale)
- This prevents old continuity data from persisting across unrelated sessions

**Detection:** A new session is identified by the absence of a `resumeSessionId` in the ContainerInput. If `sessionId` is provided (resume), keep continuity. If not (fresh), clear it. This check happens in `src/container-runner.ts` before spawning the container, since the host has write access to `data/agents/`.

### Edge Cases

| Scenario | Behavior |
|----------|----------|
| Agent doesn't call write_agent_memory within 30s | Compaction proceeds, no continuity saved (fail-open) |
| Multiple compactions in one session | Each overwrites the prior continuity section (latest wins) |
| No agent name (simple group, no compound key) | Use group folder as fallback path for continuity |
| memory.md doesn't exist | Skip injection, no error |
| Session Continuity section exceeds 1500 chars | Truncated at injection time |

## What's NOT In Scope

- **Proactive Bus Publishing (2.3)** — deferred to Ops Routing workstream
- **Auto-install of skills** — agents notify only, user runs install command
- **Token counting** — not needed; PreCompact hook fires at the right time
- **New QMD collections for session continuity** — memory.md is the persistence layer, not QMD
- **Changes to context packet size limit** — 16KB is sufficient

---

## Implementation Order

1. **Feature 2.1 (Skill Discovery)** first — independent, no risk to existing sessions
2. **Feature 2.2 (Lossless Memory)** second — touches the agent runner session lifecycle, higher risk

Features are independent and can be shipped separately.

---

## Risk Assessment

| Risk | Mitigation |
|------|------------|
| PreCompact hook blocks session | 30s timeout, fail-open |
| Skill catalog sync breaks on unusual SKILL.md | Skip malformed entries, log warning |
| QMD query timeout during skill_search | 10s timeout, return "no results" |
| Session Continuity section too large | Hard cap at 1500 chars in context-assembler |
| Agent doesn't understand self-extract prompt | Prompt is explicit with bullet format; worst case = no continuity (not harmful) |
