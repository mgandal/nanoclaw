## Overview

Phase 2 makes NanoClaw agents smarter in two ways: they can discover capabilities they don't have (Skill Discovery), and they preserve context across long sessions that would otherwise be lost to compaction (Lossless Conversation Memory).

**Scope:** Two features only. Proactive Bus Publishing (originally spec'd as 2.3) is deferred to the Ops Routing workstream per the Phase 1 spec's own recommendation.

**Dependencies resolved:**
- Agent Architecture Redesign: complete (identity.md, write_agent_memory IPC, /workspace/agents mount)
- Bus trust enforcement: already secure (sourceGroup is host-verified, unforgeable)

## Feature 2.1: Skill Discovery

### Problem

When an agent can't do something, it has no way to discover available capabilities. There are 20+ skills with rich SKILL.md files, but agents can't search them.

### Solution

A QMD-backed skill catalog generated by scanning `.claude/skills/*/SKILL.md` files on the working tree. Agents search the catalog via an MCP tool and tell the user how to install missing skills.

### Architecture

```
.claude/skills/*/SKILL.md → skill-catalog-sync.sh → data/skill-catalog/*.md → QMD collection → skill_search MCP tool → agent
```

### Components

#### 2.1.1 Skill Catalog Generator

**File:** `scripts/sync/skill-catalog-sync.sh` (new)

Host-side bash script that:
1. Scans `.claude/skills/*/SKILL.md` on the working tree (NOT git branches — skills are merged to `main`)
2. For each SKILL.md, extracts YAML frontmatter: `name` and `description` (these are the only guaranteed fields)
3. Generates `install_command` from the name: `"/{name}"` (e.g., `name: add-telegram` → `install_command: /add-telegram`)
4. Marks all working-tree skills as `installed: true` (they're on `main` by definition)
5. Writes `data/skill-catalog/{skill-name}.md` with standardized frontmatter:

```yaml
---
name: add-telegram
description: Add Telegram as a channel. Can replace WhatsApp or run alongside it.
installed: true
install_command: "/add-telegram"
---

Add Telegram as a channel. Can replace WhatsApp entirely or run alongside it.
Also configurable as a control-only channel (triggers actions) or passive channel
(receives notifications only).
```

The body is the full SKILL.md content (minus its original frontmatter), giving QMD rich text for semantic search. The `triggers` field is omitted — QMD's semantic search on the body text handles discovery without explicit trigger keywords.

**Path scoping:** Only scan `.claude/skills/*/SKILL.md` at the project root. Do NOT scan `data/sessions/` (per-session skill caches) or `groups/*/skills/` (group overrides).

**Integration:** Add as a new step in `scripts/sync/sync-all.sh`, placed **before** the QMD update step (currently step 6) so freshly generated catalog files are indexed in the same sync cycle. The catalog refresh is cheap (filesystem scan + file writes) and idempotent.

#### 2.1.2 QMD Collection

**Collection name:** `skill-catalog`
**Path:** `data/skill-catalog/`
**Pattern:** `**/*.md`

**Setup:** Add an entry to `~/.config/qmd/index.yml` (QMD's collection registry). There is no `qmd add` CLI command — collections are registered by editing this YAML file directly. One-time manual setup step, documented in the implementation plan.

Indexed during the existing `qmd update` step in sync-all.sh. Embedded during `qmd embed`.

#### 2.1.3 MCP Tool: skill_search

**File:** `container/agent-runner/src/ipc-mcp-stdio.ts` (add tool + container rebuild required)

```typescript
const SKILL_RESULTS_DIR = path.join(IPC_DIR, 'skill_results');

server.tool(
  'skill_search',
  'Search for NanoClaw capabilities and skills. Use when you need to do something ' +
    'but don\'t have the right tool. Returns matching skills with install instructions.',
  {
    need: z.string().describe('What you need to do (natural language)'),
  },
  async (args) => {
    const requestId = `skill-${Date.now()}-${Math.random().toString(36).slice(2, 8)}`;
    writeIpcFile(TASKS_DIR, {
      type: 'skill_search',
      query: args.need,
      requestId,
      groupFolder: GROUP_FOLDER,
    });
    const result = await waitForIpcResult(SKILL_RESULTS_DIR, requestId, 10000);
    if (!result) {
      return { content: [{ type: 'text', text: 'No matching skills found (or QMD unavailable).' }] };
    }
    return { content: [{ type: 'text', text: result }] };
  },
);
```

Uses the existing `waitForIpcResult(resultsDir, requestId, maxWait)` pattern (same as browser, dashboard, X tools). The `requestId` and `groupFolder` fields are required so the host can write the result to the correct path.

#### 2.1.4 Host-side IPC Handler

**File:** `src/ipc.ts` (add `skill_search` case)

New handler that:
1. Receives `{ type: 'skill_search', query, requestId, groupFolder }`
2. Makes an HTTP POST to `http://localhost:8181/mcp` (QMD's MCP endpoint) with a `query` tool call targeting the `skill-catalog` collection
3. Formats results: for each match, outputs name, description, installed status, and install command
4. Writes formatted result to `DATA_DIR/ipc/{sourceGroup}/skill_results/{requestId}.json`
5. On QMD error/timeout: writes error result distinguishing "QMD unavailable" from "no matches"

#### 2.1.5 No Auto-Install

When an agent finds an uninstalled skill, it tells the user directly via send_message:

> "I found a skill called *add-telegram* that can do this: 'Add Telegram as a channel.' To install it, run `/add-telegram`."

No approval workflow, no auto-install, no restart handling. User stays in full control.

## Feature 2.2: Lossless Conversation Memory

### Problem

Long agent sessions lose coherence when the SDK compacts (truncates) conversation context. Decisions, commitments, and learned facts vanish. The next session starts with no memory of what was discussed.

### Solution

After context compaction completes, the host injects an IPC message prompting the agent to extract and persist key context to its memory file. The context assembler then injects these items into subsequent context packets.

### Architecture

```
SDK compacts context → PreCompact hook archives transcript (existing) →
compaction completes → host writes IPC message to /workspace/ipc/input/ →
agent receives message on next turn → agent calls write_agent_memory →
host upserts ## Session Continuity section into memory.md →
next spawn: context-assembler reads memory.md → injects continuity into context packet
```

**Key design decision (from peer review):** The PreCompact hook CANNOT inject prompts the agent acts on — the agent is paused during hook execution. Instead, we use **post-compaction IPC message injection**: after compaction completes, the host writes a message file to the IPC input directory. The agent's next `drainIpcInput()` poll (every 500ms) picks it up as a new user turn. The agent is fully active and can call tools in response.

### Components

#### 2.2.1 Post-Compaction IPC Message

**File:** `container/agent-runner/src/index.ts` (extend existing PreCompact hook)

The existing `PreCompact` hook archives the transcript. We add a **flag** that the main loop checks after compaction completes:

```typescript
// In createPreCompactHook — set a flag after archiving:
let compactionJustHappened = false;

// In the PreCompact hook (after transcript archive):
compactionJustHappened = true;
return {}; // Don't block — return immediately

// In the main loop, after runQuery returns and before waitForIpcMessage:
if (compactionJustHappened) {
  compactionJustHappened = false;
  const extractMsg = path.join(IPC_INPUT_DIR, `extract-${Date.now()}.json`);
  fs.writeFileSync(extractMsg, JSON.stringify({
    type: 'message',
    text: '[SYSTEM] Your conversation context was just compacted. ' +
      'Important context may have been lost. Please call write_agent_memory ' +
      'with a concise summary of:\n' +
      '- Decisions made in this session\n' +
      '- Tasks you committed to\n' +
      '- Open questions or unresolved items\n' +
      '- Key facts the user shared that you\'ll need later\n' +
      'Use bullet points. Be concise (under 1500 chars).'
  }));
}
```

The agent picks this up on its next `drainIpcInput()` call, processes it as a normal turn, and calls `write_agent_memory`. No timeout needed — the message persists until drained.

**Manual `/compact` also triggers this** — the same PreCompact hook fires for both auto-compaction and the `/compact` slash command. The flag mechanism handles both identically.

#### 2.2.2 write_agent_memory MCP Tool + Section Upsert

**Two changes required (from peer review):**

**Change 1: Add MCP tool** — `write_agent_memory` exists as a host-side IPC handler (`src/ipc.ts:810`) but is NOT exposed as an MCP tool in `container/agent-runner/src/ipc-mcp-stdio.ts`. Agents currently cannot call it. Add:

```typescript
server.tool(
  'write_agent_memory',
  'Write or update a section of your persistent memory file. ' +
    'Content persists across sessions. Use for decisions, key facts, and session continuity.',
  {
    section: z.string().describe('Section header (e.g., "Session Continuity", "Standing Instructions")'),
    content: z.string().describe('Content for this section (bullet points recommended)'),
  },
  async (args) => {
    writeIpcFile(TASKS_DIR, {
      type: 'write_agent_memory',
      section: args.section,
      content: args.content,
    });
    return { content: [{ type: 'text', text: `Memory section "${args.section}" updated.` }] };
  },
);
```

**Change 2: Section-level upsert in IPC handler** — The current `write_agent_memory` handler (`src/ipc.ts:810`) does a full-file replacement. This is dangerous — the agent writing only a `## Session Continuity` section would clobber all other memory sections (Standing Instructions, Active Threads, etc.).

Add a **section upsert** mode: when the IPC payload includes a `section` field, the handler reads the existing `memory.md`, replaces only the matching `## {section}` block (or appends it if not found), and writes the result. Full-file replacement remains as fallback when no `section` field is provided (backwards compatibility).

```typescript
case 'write_agent_memory': {
  const memoryPath = path.join(agentDir, 'memory.md');
  if (data.section) {
    // Section upsert: read existing, replace/append section
    const existing = fs.existsSync(memoryPath)
      ? fs.readFileSync(memoryPath, 'utf-8')
      : `# ${agentName} — Memory\n`;
    const sectionHeader = `## ${data.section}`;
    const sectionRegex = new RegExp(
      `${sectionHeader}\\n[\\s\\S]*?(?=\\n## |$)`,
    );
    const newSection = `${sectionHeader}\n${data.content}`;
    const updated = existing.match(sectionRegex)
      ? existing.replace(sectionRegex, newSection)
      : `${existing.trimEnd()}\n\n${newSection}\n`;
    // Cap Session Continuity at 2000 chars
    fs.writeFileSync(memoryPath, updated);
  } else {
    // Full-file replacement (backwards compat)
    fs.writeFileSync(memoryPath, data.content);
  }
  break;
}
```

#### 2.2.3 Context Packet Injection

**File:** `src/context-assembler.ts` (add section)

Add a new section **early in the packet** (after group memory, before QMD results) so it survives truncation if the packet exceeds 16KB:

```typescript
// Read agent memory.md for Session Continuity section
if (agentName) {
  const agentMemoryPath = path.join(DATA_DIR, 'agents', agentName, 'memory.md');
  if (fs.existsSync(agentMemoryPath)) {
    const memoryContent = fs.readFileSync(agentMemoryPath, 'utf-8');
    const continuityMatch = memoryContent.match(
      /## Session Continuity\n([\s\S]*?)(?=\n## |$)/,
    );
    if (continuityMatch?.[1]?.trim()) {
      const continuity = continuityMatch[1].trim().slice(0, 1500);
      sections.push(`## Session Continuity (from prior compaction)\n${continuity}`);
    }
  }
}
```

**Budget:** 1500 characters max. Injected early (before QMD results, bus queue, classified events) so it is prioritized when the 16KB hard truncation applies. The regex uses `(?=\n## |$)` to correctly handle both mid-file and end-of-file sections.

**Guard:** Only injected when `agentName` is provided (compound groups). Simple groups without an agent name skip this section. Existing tests that don't pass `agentName` are unaffected.

#### 2.2.4 Session Start Cleanup

**File:** `src/container-runner.ts` (host-side, before container spawn)

When a new session starts (not a continuation after compaction):
- Check if `data/agents/{name}/memory.md` has a `## Session Continuity` section
- If so, remove it (the context from the prior session is stale)
- This prevents old continuity data from persisting across unrelated sessions

**Detection:** A new session is identified by the absence of `sessionId` in the ContainerInput (the field is `sessionId`, NOT `resumeSessionId` — corrected from v1). If `sessionId` is provided (resume), keep continuity. If not (fresh), clear it.

**Concurrent spawn safety:** The cleanup operates on `data/agents/{agentName}/memory.md`. If two groups share the same agent (e.g., LAB-claw and LAB-slack both spawning "marvin"), a fresh session in one group could clear continuity written by the other. This is acceptable for now — shared agents have shared memory by design. Future mitigation: scope continuity sections per group (`## Session Continuity ({groupFolder})`).

### Edge Cases

| Scenario | Behavior |
|----------|----------|
| Agent ignores the extraction prompt | No continuity saved — not harmful, just a missed opportunity |
| Multiple compactions in one session | Each overwrites the `## Session Continuity` section (latest wins) |
| No agent name (simple group, no compound key) | Skip injection — use group folder as fallback in future if needed |
| memory.md doesn't exist | Section upsert creates it with a title header |
| Session Continuity section exceeds 1500 chars | Truncated at injection time in context-assembler |
| `/compact` slash command used | Same behavior as auto-compaction (PreCompact hook sets the same flag) |
| Concurrent spawns sharing an agent | Acceptable race — shared agents have shared memory by design |
| QMD unavailable during skill_search | Return "No matching skills found (or QMD unavailable)" — not silent |

## What's NOT In Scope

- **Proactive Bus Publishing (2.3)** — deferred to Ops Routing workstream
- **Auto-install of skills** — agents notify only, user runs install command
- **Token counting** — not needed; post-compaction IPC message fires after compaction
- **New QMD collections for session continuity** — memory.md is the persistence layer, not QMD
- **Changes to context packet size limit** — 16KB is sufficient
- **Git branch scanning for skills** — skills live on `main`, not on separate branches

---

## Implementation Order

1. **Feature 2.1 (Skill Discovery)** first — independent, no risk to existing sessions
2. **Feature 2.2 (Lossless Memory)** second — touches the agent runner session lifecycle, higher risk

Features are independent and can be shipped separately.

## Risk Assessment

| Risk | Mitigation |
|------|------------|
| Skill catalog sync breaks on unusual SKILL.md | Skip malformed entries, log warning, continue |
| QMD unavailable during skill_search | 10s timeout, return descriptive error (not silent) |
| Session Continuity section too large | Hard cap at 1500 chars in context-assembler |
| Agent ignores extraction prompt | Not harmful — worst case is no continuity (fail-open) |
| Section upsert regex fails on unusual memory.md format | Falls back to append mode |
| Post-compaction IPC message not drained | Persists until next poll — no data loss, just delayed |
| Concurrent spawns race on memory.md | Acceptable for shared agents; scope by group in future |
